# /deliver

For any deliverable that would otherwise be a wall of text. Writes the full
artifact to a file, replies with a tight summary plus path. The user reads
the file, not your response.

## When to use

- Runbooks (`docs/runbooks/<name>.md`)
- Design docs / RFCs (`docs/design/<name>.md`)
- Migration plans (`docs/migrations/<name>.md`)
- Code review reports (`docs/reviews/<branch>-<date>.md`)
- Postmortems (`docs/postmortems/<incident>.md`)
- Long checklists or procedure documents

If the deliverable is under ~50 lines, skip this skill and answer inline.

## Steps

1. **Confirm the destination.** If the user named a path, use it. Otherwise
   pick from the conventions above and confirm with the user before writing
   if there is any ambiguity. Do not invent a new doc location silently.

2. **State assumptions.** Before drafting, list:
   - Audience (oncaller, reviewer, future-you, exec)
   - Scope (what's in, what's out)
   - Format constraints (template to follow, max length, sections required)

   Wait for confirmation if any of these are non-trivial.

3. **Draft to file.** Use `Write`. Structure with clear headings. Include:
   - A "Status" or "Last updated" line at the top
   - A TL;DR section first (3-5 bullets)
   - The detailed sections after

4. **Reply with only:**
   ```
   Wrote: <path>

   - <bullet 1>
   - <bullet 2>
   - <bullet 3>
   - <bullet 4>
   - <bullet 5>

   Open questions:
   - <q1>
   - <q2>
   ```

   Do not paste the document contents back. Do not narrate what's in it
   beyond the bullets.

5. **On revision**, edit the file in place via `Edit`. Reply with what
   changed in <=3 bullets. Do not re-paste the whole doc.

## Anti-patterns

- Do not stream the doc contents inline "for convenience" — that defeats
  the purpose.
- Do not create a new doc when an existing one in the same area should be
  updated. Check `docs/` first.
- Do not add a "generated by" footer or auto-sign the doc.
